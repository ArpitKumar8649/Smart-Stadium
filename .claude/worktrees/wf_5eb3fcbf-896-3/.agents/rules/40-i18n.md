# Rule 40 — Internationalisation

_Every user-facing string must be translatable. Every reply must respect the fan's language._

## The two translation layers

There are exactly two translation paths in Concourse. Do not add a third.

1. **UI strings** — static, translated at build via react-i18next JSON in `frontend/src/i18n/locales/*.json`
2. **Agent replies** — dynamic, translated at runtime by Gemini via the system prompt directive `"Respond in the language of the fan's most recent message unless they explicitly ask for another."`

Nothing else. Never call a `translate()` helper. Never bolt Cloud Translation onto anything.

## Non-seed languages

Fall back to English for UI strings; agent replies use Gemini runtime translation. UI shows a small "UI in English, Concourse understands your language" chip when the browser locale is outside the seed set.

## Rules for authors

- **Never hardcode a UI string** in a component. If you write `"Kickoff in"`, wrap it in `t('nudges.kickoffIn')`.
- **Use ICU MessageFormat** for anything with a count or gender. Not string concatenation.
- **RTL support**: always use Tailwind logical properties (`ms-*`, `me-*`, `ps-*`, `pe-*`) instead of `ml-*`, `mr-*`. Set `dir="rtl"` on `<html>` when locale is Arabic.
- **Never assume string order.** "Gate {gate} closes at {time}" is fine in English but broken in some languages — use ICU with named placeholders.

## Testing

- Every new i18n key is added to `en.json` first, then propagated to other locales
- CI has a check that all locale files have the same key set as `en.json`; missing keys fail the build
- Pseudo-locale for testing: `en-XA` (accented) — reveals text that is not wrapped in `t()`

## Language detection at boot

Order of preference:
1. Stored user preference (`localStorage.concourse.lang`)
2. `navigator.language` matched to a seed locale
3. Fallback to English

The concierge always sees the current language in its per-turn context and mirrors it in replies.
